01 — Context

What's actually changing.

Devi v2 is a fresh product concept that builds on the existing Swift codebase rather than starting from zero.

From — v1.3

Hindu Panchang almanac. Free. Fully offline. No accounts. Single-user. iOS-only.

To — v2.0

Daily spiritual companion + community. Free panchang tier + paid social tier. Backend-required for friends, synastry, content.

Carry-forward ratio ~70% of the v1.3 codebase reuses unchanged: Swiss Ephemeris, panchang engine, Living Mandala, theme system. The other 30% is new infrastructure — backend, social graph, LLM pipeline, subscriptions.
02 — From the Gemini research

Five strategic commitments.

Locked by the deep-research synthesis. Every architectural decision flows from these.

  1. Diaspora-first, English-only, USD subscription. No localization scope at v1. Indian/Nepali diaspora + Western astrology natives.
  2. Hybrid Tropical/Sidereal "archetype builder" as the central novel feature. No competitor offers this.
  3. Prescriptive astrocartography — intent → highlighted geographic zones. Not a passive map.
  4. Vedic synastry (Ashtakoot/Kuta) powering compatibility. Granular, shareable, viral.
  5. Privacy-by-design at the schema level. DPDPA + GDPR + Apple 5.1.1 force decisions in Postgres, not in policy docs.
03 — Decisions

Tier 1 — locked.

The four decisions that fork the entire architecture.

T1.1 Platform scope lockediOS-only. Continue building on existing Swift/SwiftUI codebase. Android/web deferred indefinitely.
T1.2 Migration path lockedHybrid (option C). Existing panchang stays free. New paid layers added on top. Same App Store listing.
T1.3 Backend ownership lockedBaaS — Supabase. Postgres + Apple Sign In + Realtime + Edge Functions. Re-evaluate above 50K paid users.
T1.4 Friend birth-data privacy lockedServer-side plaintext for MVP. Functionality > privacy at v1. Schema stays RLS-aware to tighten later.
03 — Decisions, cont'd

Tier 2 — locked & open.

T2.5 Auth lockedApple Sign In first. Google added later via the same Supabase abstraction.
T2.6 Content generation lockedLLM-involved, hybrid. Hand-written voice templates + LLM fills personalized variables. No visible "AI" branding.
T2.7 Offline scope lockedOwn data + panchang stay offline-capable. Friends, social, astrocartography online-only with stale-cache fallback.
T2.8 Astrocartography map tech openMapbox is the default if/when astrocartography ships. Decision deferred until v1 scope is finalized.
T2.9 Hybrid archetype model openBoth tropical & sidereal positions must be computed/stored. Picker UX deferred.

05 — Data model

Identity & birth data.

birth_data separates owner (who entered it) from subject (whose chart it represents) — friends added by birth date even before they sign up.

-- Identity
CREATE TABLE users (
  id uuid PRIMARY KEY,                -- supabase auth.users.id
  display_name text,
  created_at timestamptz,
  -- subscription state denormalized for fast reads
  -- source-of-truth lives in RevenueCat
  subscription_tier text DEFAULT 'free',  -- 'free' | 'paid'
  subscription_expires_at timestamptz
);

-- Birth data — yours, plus rows added for friends pre-signup.
CREATE TABLE birth_data (
  id uuid PRIMARY KEY,
  owner_user_id uuid REFERENCES users(id),    -- who entered it
  subject_user_id uuid REFERENCES users(id),  -- nullable for non-users
  display_name text,
  dob date,
  birth_time time,
  birth_time_unknown bool DEFAULT false,
  birth_lat double precision,
  birth_lng double precision,
  birth_place_name text,
  birth_tz_identifier text,
  created_at timestamptz
);
05 — Data model, cont'd

Friendships & archetypes.

Canonical least(a,b), greatest(a,b) ordering simplifies synastry-cache lookup. Both tropical and sidereal positions stored per user — archetype is a view over them.

-- Friend graph — single canonical row per pair.
CREATE TABLE friendships (
  user_a uuid REFERENCES users(id),  -- always least(a,b)
  user_b uuid REFERENCES users(id),  -- always greatest(a,b)
  status text,                          -- 'pending' | 'accepted' | 'blocked'
  initiated_by uuid REFERENCES users(id),
  created_at timestamptz,
  PRIMARY KEY (user_a, user_b)
);

-- One archetype per user for v1 (multi-archetype deferred).
CREATE TABLE archetypes (
  user_id uuid PRIMARY KEY REFERENCES users(id),
  -- per-planet system selection. jsonb for extensibility.
  -- e.g., { "sun": "tropical", "moon": "sidereal", ... }
  system_mapping jsonb DEFAULT '{}',
  -- precomputed positions in BOTH systems
  tropical_positions jsonb,                -- { sun: {sign,deg}, moon:..., ...}
  sidereal_positions jsonb,                -- same shape
  computed_at timestamptz
);
05 — Data model, cont'd

Synastry & daily content.

Both are caches keyed by deterministic inputs. Synastry by user-pair; daily content by (user, date). Astrocartography schema deferred until map tech is locked.

-- Synastry cache (Ashtakoot/Kuta + tropical aspects)
CREATE TABLE synastry_results (
  user_a uuid,
  user_b uuid,
  kuta_scores jsonb,                -- 8-factor Vedic
  tropical_aspects jsonb,           -- western synastry layer
  llm_summary text,                  -- "communication clashes but soul missions align"
  computed_at timestamptz,
  PRIMARY KEY (user_a, user_b)
);

-- Daily content cache — one row per (user, date)
CREATE TABLE daily_content (
  user_id uuid,
  content_date date,                -- user's local date
  ritual_prompt text,               -- Living Mandala framing
  transit_summary text,             -- "Mars squaring natal Venus today"
  hybrid_insight text,              -- pulls from archetype mapping
  llm_model text,
  generated_at timestamptz,
  PRIMARY KEY (user_id, content_date)
);

-- astrocartography_intent — schema deferred until map decision
06 — Privacy

RLS-aware schema, even in permissive MVP.

"Privacy not really important for MVP" is a defensible call — but the shape of the system should still be RLS-aware so we can tighten the policy without rewriting the data layer.

  • Enable RLS on every table from day one. Default policies allow auth.uid() = owner for read/write on own rows.
  • Friendship rows readable by both user_a and user_b.
  • Synastry results readable by both parties of the pair.
  • The MVP can run permissive policies while the structure stays correct — flipping to strict is a policy change, not a migration.
Why this matters DPDPA 2023 has extra-territorial reach for any service offered to users in India. Even if v1 doesn't enforce strict privacy, the schema shape determines whether tightening later costs an afternoon or a quarter.

08 — Auth

Anonymous-first.

Apple 5.1.1 prohibits gating basic app function behind PII collection. So the free tier works without sign-in; the auth wall sits at the social or paid boundary.

  1. App launches → check Supabase session. Anonymous allowed.
  2. Free tier works without sign-in. Panchang, today's date, sample content — all visible immediately. No birth-data wall.
  3. Sign-in is triggered by adding friends OR subscribing → Sign in with Apple → Supabase creates users row.
  4. Birth data prompt comes after sign-in, contextualized: "Add your birth details to see your chart."
  5. Google added later via the same Supabase auth interface. No client-side change.
09 — Content

LLM pipeline — and avoiding the Co-Star trap.

The Co-Star failure mode is users feeling read by an algorithm. Mitigations are architectural, not stylistic.

  • Hand-written voice templates define structure and tone. LLM fills variables, doesn't generate freely.
  • LLM operates on structured astrological context, not freeform prompts. Edge Function computes today's transits → builds structured input → Claude generates prose constrained by template + voice guide.
  • Voice guide doc loaded into every prompt as system message. Defines vocabulary, prohibited phrases, sacred-vs-casual register.
  • Prompt caching on the system message + voice guide → ~10× cost reduction.
  • No "AI" branding visible to user. Don't say "Powered by Claude." Just deliver content.
Critical path The voice guide is on the critical path. Without it — or without an astrologer partner to author it — content quality collapses to Co-Star-grade output regardless of model choice.
09 — Content, cont'd

LLM cost economics.

With prompt caching, daily content is comfortably under 30% margin against a $10/mo subscription.

Per-user-per-day$0.01 – $0.03 · with prompt caching on voice guide + system message.
10K paid users$100 – $300 / day · daily content alone.
Monthly LLM spend$3K – $9K / month · at 10K paid users.
Margin against $10/mo sub< 30% · LLM is the dominant variable cost; everything else (Supabase, RevenueCat) is materially smaller.
Sanity check Synastry + astrocartography insights are episodic, not daily — they don't materially change the per-user-per-day cost ceiling unless we let the user re-roll the LLM output on demand.

11 — Open

Five forks before code starts.

Roughly priority-ordered. Item 1 is the difference between an 8-week and a 14-week build.

  1. Astrocartography v1 vs v1.1. Most computationally + UX-novel surface. If v1 → pick map tech, design intent UI. If deferred → ship the diaspora wedge faster.
  2. Hybrid archetype UI. Does v1 even have a picker, or just show both systems side-by-side? Latter is dramatically simpler.
  3. LLM voice / writer partnership. Astrologer partner for voice guide and seed templates. Critical path — content quality determines premium feel.
  4. Subscription feature split. Free = panchang + own chart + daily mandala. Paid = friends/synastry + astrocartography + deep daily insight. Confirm before wiring entitlements.
  5. Push notification strategy. Daily ritual nudge? Time-window alerts (Brahma Muhurta)? Friend activity? Choose 1–2 for v1; more is spam.

13 — Risks

Risks we're architecting around.

Apple 4.3 — spam

Generic horoscope apps get rejected. Lifestyle-companion framing required. Living Mandala + hybrid archetype + prescriptive astrocartography are the differentiators that satisfy "elevate beyond a repackaged website."

Apple 5.1.1 — data collection

Birth data cannot gate basic app function. Already addressed by anonymous-first auth flow. Birth-data prompts come after sign-in, contextualized.

DPDPA 2023 / GDPR

Even with permissive MVP privacy posture, schema must be RLS-aware so tightening later doesn't require migration. Extra-territorial reach matters once we have any India-resident user.

Traditionalist backlash

State Swiss Ephemeris + Lahiri ayanamsa prominently. Frame hybrid as "two lenses on the same sky," not "synthesis of contradictory systems."

Content credibility

LLM voice without an astrologer partner risks Co-Star-style "feels generic" complaints. Voice guide partnership is on the critical path. Mitigation: hand-written templates, no visible AI branding.

Subscription rejection / refund cycles

Nebula was hammered for deceptive billing. RevenueCat keeps the receipt-validation flow audit-ready. Free tier must remain genuinely useful — no dark-pattern paywall escalation.
